\chapter{Introduction}

\section{Contexte général}
Une petite entreprise a développé une application web. Cette application, développée rapidement, n'a jamais été auditée pour sa sécurité. Nous avons été mandatés comme consultants pour réaliser un audit de sécurité complet de cette application et proposer des mesures concrètes de renforcement.

\section{Choix de l'application cible}
Nous avons choisi l'application OWASP Juice Shop, une plateforme intentionnellement vulnérable développée en Node.js avec le framework Express. Cette application présente une large gamme de vulnérabilités courantes, classées par niveau de difficulté (de 1 à 6 étoiles).

\section{Objectifs de l'audit}
Les objectifs de cet audit sont :
\begin{itemize}
    \item Identifier les vulnérabilités présentes dans l'application
    \item Démontrer les techniques d'exploitation avec des payloads exacts
    \item Documenter les impacts de chaque faille
    \item Proposer des recommandations de correction
    \item Produire un rapport technique professionnel
\end{itemize}

\chapter{Présentation de l'application}

\section{Fonctionnalités principales}
OWASP Juice Shop simule une boutique en ligne avec les fonctionnalités suivantes :
\begin{itemize}
    \item Inscription et connexion des utilisateurs
    \item Catalogue de produits
    \item Panier d'achat
    \item Gestion de compte utilisateur
    \item Section administrateur
    \item Système de feedback et commentaires
\end{itemize}

\section{Architecture technique}
L'application repose sur une architecture moderne :
\begin{itemize}
    \item Backend : Node.js avec Express.js
    \item Base de données : SQLite (par défaut)
    \item Frontend : Angular
    \item API REST pour la communication entre le frontend et le backend
\end{itemize}

\section{Architecture de l'environnement de test}
L'audit a nécessité deux phases d'installation afin de garantir la couverture de l'intégralité du périmètre de test.

\subsection{Phase 1 : Installation Conteneurisée (Docker)}
Une première instance a été déployée via l'image officielle bkimminich/juice-shop. Cette phase a permis d'effectuer un premier passage sur une majorité des vulnérabilités (injections, XSS, etc.).

\subsection{Phase 2 : Installation Native (XAMPP)}
Suite au constat de challenges inaccessibles (liés à des RCE, écriture de fichiers arbitraires et accès au système de fichiers hôte), une seconde instance a été déployée nativement sur un serveur XAMPP. Ce déploiement a été motivé par le fait que l'isolation réseau d'un conteneur Docker bloque certaines attaques qui nécessitent une interaction avec l'hôte. L'application a été configurée en mode "unsafe" (via la variable NODE\_ENV=unsafe) afin de s'assurer que l'ensemble des challenges "dangereux" (nécessitant des accès systèmes) soient activés.

\section{Points d'entrée}
Les principaux points d'entrée de l'application sont :
\begin{itemize}
    \item Formulaire de connexion
    \item Formulaire d'inscription
    \item Barre de recherche
    \item Formulaire de feedback
    \item API REST
    \item Paramètres URL
\end{itemize}

\section{Outils utilisés}
Les outils suivants ont été utilisés pour réaliser cet audit :
\begin{itemize}

L'audit a suivi une approche méthodique :
\begin{enumerate}

    \item Identification des points d'entrée
    \item Test des vulnérabilités par catégorie

    \item Validation des exploits
\end{enumerate}

\chapter{Vulnérabilités identifiées}

\section{Challenges ⭐ (Très Facile)}

\subsection{Score Board}
\begin{vulnerability}{Score Board}{A05:2021 - Security Misconfiguration}{⭐}
    \subsubsection*{Description}
    Trouver la page Score Board soigneusement cachée.
    
    \subsubsection*{Technique d'exploitation}
    \begin{enumerate}
        \item Clic droit sur la page d'accueil > Inspecter > Sources
        \item Rechercher `path` ou `score` dans `main.js`
        \item Accéder à l'URL : `http://[ip]:3000/#/score-board`
    \end{enumerate}
    
    \begin{figure}[H]
        \centering
        \includegraphics[width=0.8\textwidth]{score_board.png}
        \caption{Accès au Score Board}
    \end{figure}
    
    \subsubsection*{Impact sur la sécurité}
    \begin{itemize}
        \item Sévérité: Basse
        \item Impact: Exposition de la liste des vulnérabilités
    \end{itemize}
    
    \subsubsection*{Recommandations de correction}
    \begin{recommendation}{Moyenne}
        Restreindre l'accès au score board aux seuls administrateurs.
    \end{recommendation}
\end{vulnerability}

    \subsubsection*{Technique d'exploitation}
    \begin{payload}
<iframe src="javascript:alert('xss')">
    \end{payload}
    \begin{enumerate}
        \item Cliquer sur l'icône de recherche
        \item Saisir le payload dans la barre de recherche
        \item Appuyer sur Entrée
    \end{enumerate}
    
    \begin{figure}[H]
        \centering
        \includegraphics[width=0.8\textwidth]{dom_xss.png}
        \caption{DOM XSS dans la barre de recherche}
    \end{figure}
    
    \subsubsection*{Impact sur la sécurité}
    \begin{itemize}
        \item Sévérité: Moyenne
        \item Impact: Exécution de code JavaScript dans le navigateur de la victime
    \end{itemize}
    
    \subsubsection*{Recommandations de correction}
    \begin{recommendation}{Critique}
        Échapper toutes les données utilisateur avant de les insérer dans le DOM.
    \end{recommendation}
\end{vulnerability}

\subsection{Unsigned JWT}
\begin{vulnerability}{Unsigned JWT}{A02:2021 - Cryptographic Failures}{⭐⭐⭐⭐⭐}
    \subsubsection*{Description}
    Forger un token JWT non signé pour usurper l'identité d'un utilisateur.
    
    \subsubsection*{Technique d'exploitation}
    \begin{enumerate}
        \item Se connecter et récupérer le token JWT
        \item Décoder le token
        \item Changer l'email en `jwtn3d@juice-sh.op`
        \item Changer l'algorithme (`alg`) de `RS256` à `none`
        \item Réencoder le token et l'utiliser
    \end{enumerate}
    
    \begin{figure}[H]
        \centering
        \includegraphics[width=0.8\textwidth]{unsigned_jwt.png}
        \caption{Forgery de JWT non signé}
    \end{figure}
    
    \subsubsection*{Impact sur la sécurité}
    \begin{itemize}
        \item Sévérité: Critique
        \item Impact: Usurpation d'identité complète
    \end{itemize}
    
    \subsubsection*{Recommandations de correction}
    \begin{recommendation}{Critique}
        Vérifier systématiquement la signature des JWT et ne jamais accepter l'algorithme `none`.
    \end{recommendation}
\end{vulnerability}

\chapter{Recommandations générales}

\section{Contre-mesures correctives}

\subsection{Injections}
\begin{itemize}
    \item Utiliser systématiquement des requêtes paramétrées
    \item Éviter la concaténation de chaînes pour construire des requêtes SQL/NoSQL
    \item Valider et nettoyer toutes les entrées utilisateur
\end{itemize}

\subsection{XSS}
\begin{itemize}
    \item Échapper les données utilisateur selon le contexte (HTML, JavaScript, CSS, URL)
    \item Implémenter une Content Security Policy (CSP) stricte
    \item Utiliser des frameworks qui échappent automatiquement les données (Angular, React)
\end{itemize}

\subsection{Authentification}
\begin{itemize}
    \item Implémenter un verrouillage de compte après plusieurs tentatives échouées
    \item Utiliser l'authentification à deux facteurs (2FA)
    \item Stocker les mots de passe avec un algorithme de hachage fort (bcrypt, Argon2)
    \item Vérifier systématiquement la signature des JWT et ne jamais accepter `alg: none`
\end{itemize}

\subsection{Contrôle d'accès}
\begin{itemize}
    \item Vérifier les autorisations pour chaque requête
    \item Utiliser des UUID au lieu d'identifiants séquentiels
    \item Implémenter le principe du moindre privilège
\end{itemize}

\subsection{Configuration}
\begin{itemize}
    \item Désactiver les messages d'erreur détaillés en production
    \item Supprimer les fonctionnalités de débogage
    \item Mettre à jour régulièrement les dépendances
\end{itemize}

\section{Recommandations prioritaires}
\begin{enumerate}
    \item \textbf{Critique} : Corriger immédiatement toutes les vulnérabilités permettant un accès non autorisé ou une exécution de code à distance
    \item \textbf{Élevée} : Mettre en place une politique de sécurité des contenus (CSP) et valider toutes les entrées utilisateur
    \item \textbf{Moyenne} : Implémenter un mécanisme de verrouillage de compte et utiliser des hachages forts pour les mots de passe
\end{enumerate}

\chapter{Conclusion}

\section{Synthèse}
Cet audit a permis d'identifier et de documenter une large gamme de vulnérabilités dans l'application OWASP Juice Shop. Ces vulnérabilités couvrent toutes les catégories de l'OWASP Top 10, des injections SQL aux attaques SSRF en passant par les contournements d'authentification.

\section{Recommandations globales}
Pour améliorer la sécurité de l'application, il est recommandé de :
\begin{itemize}
    \item Corriger toutes les vulnérabilités identifiées en priorité
    \item Mettre en place des tests de sécurité réguliers
    \item Former les développeurs aux bonnes pratiques de sécurité
    \item Utiliser des outils d'analyse de sécurité (SAST, DAST)
\end{itemize}

\section{Perspectives}
Ce travail pourrait être complété par :
\begin{itemize}
    \item Un audit de sécurité plus approfondi
    \item Des tests de performance
    \item Une analyse de la sécurité du code source
\end{itemize}
